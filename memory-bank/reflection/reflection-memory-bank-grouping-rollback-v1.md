# Reflection: Memory Bank — попытка группировки документов (отменено) — v1

Дата: 2025-12-20

## Summary
Попробовали сделать группировку `memory-bank/` по папкам (условно “process” и “project”), но результат оказался неудобным и вызвал путаницу. В итоге **полностью откатили структуру** обратно в плоский вид “как было”.

## What went well
- Быстро выяснили реальную стоимость изменения структуры: затрагиваются не только документы, но и **внешние правила/инструменты**, которые ожидают фиксированные пути.
- Откат сделали аккуратно: вернули файлы на места и вычистили новые пути из ссылок.

## Challenges
- **Совместимость с существующими ожиданиями**: часть документации/инструментов (в т.ч. описания команд) опирается на конкретные пути (`memory-bank/tasks.md`, `memory-bank/activeContext.md`, и т.д.).
- **Технические ограничения окружения** при массовых заменах (inplace-замены в некоторых файлах; ограничения на “ARG_MAX” в sandbox) — добавили трения.
- Главная проблема — **UX**: “группировка ради группировки” не дала ощутимого выигрыша, а когнитивную нагрузку увеличила.

## Lessons learned
- Для `memory-bank/` важнее **стабильность путей и простота навигации**, чем иерархия папок.
- Если хочется порядка — лучше начинать с:
  - строгого `README.md` (индекс/карта),
  - единых соглашений по именованию,
  - “ссылочной” навигации между документами,
  а не с физического перемещения файлов.

## Process improvements
- Любые изменения структуры документации делать как отдельную задачу с:
  - критериями успеха (что стало лучше),
  - планом совместимости (что ожидает старые пути),
  - явной “точкой отката”.

## Next steps
- Оставляем плоскую структуру `memory-bank/` как есть.
- Если снова вернёмся к теме порядка — обсуждаем сначала **зачем** и **какой выигрыш** хотим получить (скорость навигации, меньше дублей, яснее “что читать”), и только потом выбираем механизм.
